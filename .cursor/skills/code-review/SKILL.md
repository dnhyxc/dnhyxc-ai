# 代码审查（code-review）Skill

目标：在有限上下文下快速定位高风险问题，给出可落地的修复建议，并用一致的输出格式让作者能直接行动。

## 审查流程（按顺序）

### 1) 先明确“审查对象”与“变更边界”

- 优先基于用户提供的：文件路径、`git diff`、提交范围、PR 描述、报错日志。
- 若信息不足，优先用搜索定位：变更入口点、调用链、配置开关、影响面（API/DB/权限/缓存/异步任务）。

### 2) 按风险优先级检查

- **正确性**：空值/边界条件、并发/竞态、状态一致性、异常路径、回滚语义、幂等性。
- **安全性**：输入校验、注入（SQL/XSS/命令）、鉴权与越权、敏感信息泄漏、SSRF/开放重定向、文件上传风险。
- **性能**：N+1、重复计算、无界循环/递归、过度渲染、阻塞 IO、缓存策略、算法复杂度。
- **可维护性**：职责拆分、命名一致性、重复代码、模块边界、可测试性、可观测性（日志/指标）。
- **兼容性与稳定性**：类型收敛、序列化/反序列化、时间与时区、浮点精度、依赖升级风险。

### 3) 找“必须改”的点，再给“可选优化”

- 先收敛到 3-8 个最高信号问题，避免面面俱到。
- 每个问题都要包含：**影响**、**证据（具体代码位置/路径/片段）**、**建议**、**最小修复**。

### 4) 给出测试建议（只提最小必要集）

- 回归路径：变更影响到的主流程。
- 关键边界：空值、极大/极小、重复请求、并发、权限角色。
- 若已有测试：指出应补的用例类型（单测/集成/E2E），不强行铺开。

## 输出格式（必须使用）

按以下结构输出审查结果：

```markdown
## 总结

- **结论**：通过 / 需修改后通过 / 不建议合入
- **主要风险**：1-3 条（只写最高风险）

## 必须修改（P0/P1）

1. **[P0] 标题**
   - **影响**：
   - **证据**：`path:line-range` 或引用关键代码片段
   - **建议**：
   - **最小修复**：

## 建议优化（P2）

- **标题**
  - **原因**：
  - **建议**：

## 测试建议

- [ ] 用例 1
- [ ] 用例 2
```

## 严重程度定义（固定）

- **P0**：安全漏洞、数据损坏、线上崩溃、权限越权、不可恢复的错误。
- **P1**：高概率 bug、重要边界缺失、明显的并发问题、性能显著退化。
- **P2**：可维护性/一致性/风格问题，或中低收益优化。

## 代码审查准则（可操作）

- **指出“哪里错”要具体**：用路径与局部片段，不要只说“有问题”。
- **修复建议要最小化**：优先给一个最小可行补丁方向，而不是大改架构。
- **避免过度主观**：风格类建议放 P2，除非影响正确性/安全。
- **不要凭空假设**：对外部系统/接口/配置不确定时，明确“不确定点”并给验证方式（例如查调用方、查 schema、跑用例）。

## 常见检查清单（按需选用）

- **输入与输出**：是否做了 schema 校验？是否返回一致的错误结构？
- **错误处理**：是否吞错？是否误用 `catch` 造成静默失败？
- **资源释放**：连接/订阅/定时器/事件监听是否正确清理？
- **权限与审计**：是否需要记录操作日志？是否需要鉴权检查？
- **并发与幂等**：重复提交/重试是否会重复写入？
