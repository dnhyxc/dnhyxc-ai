# 产品向文档：`project-guide.md` 与 `project-update-info.md`

本轮改动若包含**用户可感知的新功能、体验优化或行为修复**，在写完专题实现文后，须**同步**更新下列两份文档（仍属 `docs/` 内，本 Skill 允许编辑）。

应用内 `/project-guide`、`/update-info` 由前端结构化数据驱动。维护姊妹稿后须**同轮**按 [`product-pages-sync.md`](./product-pages-sync.md) 同步 `updateInfoSections.ts`、`projectGuideSections.ts` 及对应 `*EnOverlay.ts`（共 4 个文件）。

---

## 1. 硬性约束：正文不得出现文件路径

在 **`docs/project-guide.md`** 与 **`docs/project-update-info.md`** 的正文（含列表项、示例说明）中：

| 禁止 | 允许 |
|------|------|
| 仓库相对/绝对路径（`apps/...`、`docs/...`、`packages/...`） | 产品模块名（知识库、对话、英语学习） |
| 文件名、目录名、`.ts`/`.tsx` 路径 | 路由 **路径片段**（如 `/chat`、`/update-info`）— 面向用户的外链/书签 |
| 「见 xxx.md」「见 § 某专题文」类开发索引 | 功能名、按钮名、设置项名（与 UI 一致） |
| 环境变量名、`Info.plist` 等工程配置名（除非产品设置页已暴露同名文案） | 技术术语 + 括号中文（如 SSE、RAG、ATS）— 与现有文风一致 |

**专题实现文**（`docs/knowledge/...` 等）仍须保留路径与代码来源标注；产品向两份文档单独遵守上表。

## 2. `docs/project-update-info.md` 格式

- 文首保留产品级导语（强调可感知能力、**正文不列工程路径**）。
- 按 **`## N. 章节标题`** 分域（与现有 §1–§24 风格一致）；新能力归入**已有章节**优先，确实无合适章节再追加新 `##` 节。
- 每条能力一条 bullet，推荐形态：

```markdown
- **<功能短标题>**（更新：YYYY-MM-DD）：<1～3 句，用户能感知的变化；可引用其它 § 编号，如「与 **§6** 衔接」>。
```

- **日期**：用当天或用户指定的发布日期；同一轮多条可用同一日期。
- **语气**：产品发布说明，不写「修改了某某文件」。
- **不写**：diff 清单、函数名、内部 Store 名（除非 UI 上可见同名文案）。

---

## 3. `docs/project-guide.md` 格式

- 面向**使用教程**：场景、步骤、注意事项、示例提问结构。
- 大结构为 **`## N. 章节`** + **`### N.x 小节`**；新功能优先并入相关章（如知识库 → §5，英语学习 → §13），避免堆在文末孤立节。
- 列表、步骤用「进入某某」「点击某某」等**操作语义**；可用 **`/route`** 形式写公开路由。
- **禁止**在教程正文中写「实现见 docs/…」「源码在 apps/…」；深度实现留给 `docs/README.md`（§14 仅保留一句「克隆仓库后查阅开发文档总索引」，**不写** `docs/` 下具体文件名）。
- 与 `project-update-info.md` **姊妹维护**：教程写「怎么用」，更新信息写「新增/优化了什么」；避免大段复制粘贴。

---

## 4. 如何判断「是否需要写入产品向文档」

| 写入 update-info + 视情况写 guide | 仅写专题实现文即可 |
|-----------------------------------|-------------------|
| 新页面、新入口、新设置项 | 纯重构、无行为变化 |
| 用户可见的 Bug 修复（流式、图片不显示等） | 内部日志、注释、类型收紧 |
| 交互/文案/性能可感知优化 | 仅测试、CI、构建脚本 |
| 新 API 导致的新产品能力 | 仅开发者工具链 |

不确定时：**默认补一条简短的 update-info**；若涉及操作步骤，再在 guide 增加小节或段落。
